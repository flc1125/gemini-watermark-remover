# 强证据小尺寸自适应候选清理资格设计

## 背景

`severe-10` 的真实水印约为 `40x40`，位置为 `(611, 902)`。旧搜索只能看到更大的候选，因而把局部小星误当成完整大模板的副星。补充 40px 强证据搜索后，定位已正确，但候选来源为 adaptive、边距为 82px，不符合现有 known-48 清理门只接受标准 32px 或 96px 边距的条件。

当前正确定位后的残留指标为：空间 `0.067`、梯度 `0.372`、halo `2.09`、瑕疵成本 `0.388`。一次临时单变量实验让该候选进入现有清理链后，指标变为：空间 `0.128`、梯度 `0.277`、halo `0.83`、瑕疵成本 `0.309`。实验代码已还原。

## 目标

- 仅让经过强证据门确认的约 40px adaptive 候选使用现有 known-48 边缘清理链。
- 不修改普通 48px/96px 候选、不放宽全局边距、不新增清理算法。
- 保持 Top-N 的 best-effort 行为和瑕疵信号反馈。
- 36 样本回归不得低于当前 `clean=19`、干净恢复 `15/16`、灾难性误处理 `0`。

## 方案比较

### 方案 A：显式 provenance 标记（采用）

自适应检测只有在 40px 候选同时满足现有强证据门时，才输出 `strongUndersizedMatch=true`。候选构建过程保留该标记；清理门按标记、尺寸范围和来源放行。

优点是语义明确，清理层不需要重新推断检测原因，也不会依赖字符串拼接。缺点是需要确认标记能沿候选数据流完整传递。

### 方案 B：按 source 字符串放行

允许 `adaptive...` 且尺寸为 40px 的来源进入清理链。

实现最短，但来源字符串会随增益、微调和修复阶段变化，容易把弱证据 adaptive 候选一起放行，因此不采用。

### 方案 C：把 82px 边距加入 known-48 catalog

将 `(40, 82, 82)` 当成固定 catalog 锚点。

这会把单一样本观察泛化为官方尺寸规则，而且无法表达“只有强证据时才可信”，因此不采用。

## 设计

### 候选标记

40px 补充搜索继续使用当前条件：基础层级至少 80px，候选空间分数至少 `0.9`、梯度分数至少 `0.75`，并比常规候选置信度高至少 `0.12`。只有该分支最终获胜时，检测结果携带 `strongUndersizedMatch=true`。

该字段通过自适应候选构建进入 trial provenance，后续增益和细化候选必须保留 provenance。

### 清理资格

known-48 清理门新增一条独立资格：

- provenance 中 `strongUndersizedMatch === true`；
- 最终位置尺寸在 `38..42px`；
- 非 preview-anchor；
- 来源仍属于 adaptive 路径。

满足该资格时可以绕过标准 32px/96px 边距检查，但仍复用现有残留、空间漂移、梯度改善、halo 和近黑比例验收条件。清理候选不满足现有质量门时保持原结果。

### 数据流

`detectAdaptiveWatermarkRegion` 识别强小尺寸候选 → adaptive trial 保存 provenance → Top-N 选择和细化保留 provenance → `shouldUseKnown48EdgeCleanup` 识别显式资格 → 现有 edge cleanup / flat fill / luma edge 逐级尝试 → 现有质量门决定是否采用。

## 测试

1. 自适应检测测试确认强 40px 候选输出显式标记，普通候选不输出。
2. 清理门测试确认带标记的 38–42px adaptive 候选可用；无标记、尺寸越界或非 adaptive 候选不可用。
3. `severe-10` 实际输出应保持 40px 定位，梯度、halo 和瑕疵成本较当前基线改善，不出现大号星形误处理。
4. 运行 36 样本对比、419 样本回归、完整单元测试和生产构建。

## 非目标

- 不继续调 40px alpha 模板或全局增益。
- 不新增 inpaint、模糊或专用填充算法。
- 不把 82px 边距写入 Gemini 官方尺寸 catalog。
- 不改变瑕疵信号和 best-effort 返回契约。
